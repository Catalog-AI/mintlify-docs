The Catalog API provides endpoints for product search, discovery, and commerce intelligence. All endpoints return JSON responses.

## Base URL

```
https://api.getcatalog.ai
```

## Authentication

All requests require an API key in the `x-api-key` header. See [Authentication](/v1/authentication) for details.

## Endpoints

### Listings

| Endpoint | Method | Description |
|----------|--------|-------------|
| [`/v1/vendors`](/v1/api-reference/endpoints/listings/get-vendors) | GET | List all vendors with product counts |
| [`/v1/collections`](/v1/api-reference/endpoints/listings/get-collections) | POST | List collections for a vendor |
| [`/v1/listings`](/v1/api-reference/endpoints/listings/get-products) | POST | List products from a vendor or collection |

### Products

| Endpoint | Method | Description |
|----------|--------|-------------|
| [`/v1/products`](/v1/api-reference/endpoints/products/retrieve-products) | POST | Start async batch processing of product URLs (1-1000 URLs) |
| [`/v1/products/{execution_id}`](/v1/api-reference/endpoints/products/get-execution-status) | GET | Check status and retrieve results for a batch processing job |

### Search

| Endpoint | Method | Description |
|----------|--------|-------------|

| [`/v1/agentic-search-mini`](/v1/api-reference/endpoints/search/agentic-search-mini) | POST | Fast AI search returning up to 10 products |

### Affiliate

| Endpoint | Method | Description |

### Usage

| Endpoint | Method | Description |
|----------|--------|-------------|

## Response format

Response structures vary by endpoint:

**Listings endpoints** (vendors, collections, listings) return:
```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "page_size": 10,
    "total_items": 100,
    "total_pages": 10,
    "has_next": true,
    "has_prev": false
  },
  "meta": {}
}
```

**Search endpoints** (agentic-search, agentic-search-mini) return:
```json
{
  "products": [...],
  "meta": {
    "totalItems": 100,
    "urls": [...]
  }
}
```

**Product execution status** returns:
```json
{
  "status": "completed",
  "results": {
    "products": [...],
    "meta": {
      "total_requested": 100,
      "total_successful": 98,
      "total_failed": 2
    }
  },
  "pagination": {
    "page": 1,
    "limit": 50,
    "total_items": 100,
    "total_pages": 2,
    "has_next": true
  }
}
```

## Error handling

Errors return appropriate HTTP status codes with error details. The error format varies:

**Standard error responses** (400, 401, 402, 403, 404, 409, 429, 500):
```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error description",
    "request_id": "req_..."
  }
}
```

**Execution status errors** return a simple string:
```json
{
  "status": "failed",
  "error": "The operation failed. Please retry or contact support with this execution ID."
}
```

| HTTP Status | Meaning | Description |
|-------------|---------|-------------|
| `400` | Bad Request | Invalid request parameters or malformed JSON |
| `401` | Unauthorized | Missing or invalid API key |
| `402` | Payment Required | Insufficient credits to complete the request |
| `403` | Forbidden | API key lacks required permissions |
| `404` | Not Found | Requested resource does not exist |
| `409` | Conflict | Resource state conflict |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Server-side error occurred |

All error responses include a `Request-ID` header that matches the `request_id` in the response body. See [Error Codes](/resources/error-codes) for the complete reference.

## Rate limits

Default policy: 50 requests per 4-second sliding window, counted per IP with a global fallback.

See [Rate Limits](/resources/rate-limits) for headers, what `429` means, and pacing guidance.

## Pagination

Most list endpoints support pagination using query parameters:

**GET endpoints** (vendors, usage) use query string parameters:
- `page`: Page number (1-indexed, default: 1)
- `page_size`: Items per page (default varies, max: 100)

**POST endpoints** (listings, collections, agentic-search) include pagination in the request body:
- `page`: Page number (required for listings/collections)
- `page_size`: Items per page (optional, default varies)

**Execution status endpoint** (`/v1/products/{execution_id}`) uses different parameter names:
- `page`: Page number (1-indexed, default: 1)
- `limit`: Items per page (default: 50, max: 100)

All paginated responses include a `pagination` object in the response with:
- `page`: Current page number
- `page_size` or `limit`: Items per page (depends on endpoint)
- `total_items`: Total number of items
- `total_pages`: Total number of pages
- `has_next`: Whether there is a next page
- `has_prev`: Whether there is a previous page

<Note>
The `total_items` count may be capped for performance depending on the endpoint. See [Pagination Guide](/v1/guides/pagination) for implementation patterns.
</Note>
