The Catalog API uses standard HTTP status codes and returns detailed error information to help you troubleshoot issues quickly.

## Error response format

All error responses follow a consistent structure:

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error description",
    "details": {
      "parameter": "field_name",
      "expected": "expected_type",
      "received": "actual_type"
    }
  }
}
```

## HTTP status codes

| Status Code | Meaning | Description |
|-------------|---------|-------------|
| `200` | Success | Request completed successfully |
| `400` | Bad Request | Invalid request parameters or malformed JSON |
| `401` | Unauthorized | Missing or invalid API key |
| `403` | Forbidden | API key lacks required permissions |
| `404` | Not Found | Requested resource does not exist |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Server-side error occurred |

## Error codes reference

### Authentication errors

| Code | HTTP Status | Description | Resolution |
|------|-------------|-------------|------------|
| `UNAUTHORIZED` | 401 | Missing or invalid API key | Verify your API key is correct and included in the `x-api-key` header |
| `FORBIDDEN` | 403 | API key access denied | Contact support to verify your API key permissions |

### Request validation errors

| Code | HTTP Status | Description | Resolution |
|------|-------------|-------------|------------|
| `INVALID_PARAMETER` | 400 | Invalid request parameter | Check the parameter requirements in the endpoint documentation |
| `MISSING_PARAMETER` | 400 | Required parameter missing | Include all required parameters in your request |
| `INVALID_JSON` | 400 | Malformed JSON body | Validate your JSON syntax before sending |
| `INVALID_URL` | 400 | Invalid URL format | Ensure URLs are properly formatted and encoded |

### Resource errors

| Code | HTTP Status | Description | Resolution |
|------|-------------|-------------|------------|
| `NOT_FOUND` | 404 | Resource not found | Verify the endpoint URL and any resource IDs |
| `PRODUCT_NOT_FOUND` | 200 | Product could not be retrieved | The product URL may be invalid, unavailable, or from an unsupported vendor |
| `VENDOR_NOT_SUPPORTED` | 200 | Vendor is not supported | Check the list of supported vendors or contact support |

### Rate limiting errors

| Code | HTTP Status | Description | Resolution |
|------|-------------|-------------|------------|
| `RATE_LIMITED` | 429 | Too many requests in a short window | Respect `Retry-After`, pace requests using `X-RateLimit-Remaining`, and back off (exponential recommended). Contact support if you need higher limits. |

### Server errors

| Code | HTTP Status | Description | Resolution |
|------|-------------|-------------|------------|
| `INTERNAL_ERROR` | 500 | Unexpected server error | Retry the request; if persistent, contact support |
| `TIMEOUT` | 500 | Request timeout | Retry with a smaller batch size or simpler query |
| `DB_TIMEOUT` | 500 | Database query timeout | Simplify your query or retry later |
| `DB_ERROR` | 500 | Database error | Retry the request; if persistent, contact support |

## Handling errors

### Basic error handling

<CodeGroup>

```javascript JavaScript
const response = await fetch('https://api.getcatalog.ai/v1/search', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-api-key': 'YOUR_API_KEY'
  },
  body: JSON.stringify({ query: 'black dress' })
});

if (!response.ok) {
  const error = await response.json();
  
  switch (response.status) {
    case 401:
      console.error('Authentication failed:', error.message);
      break;
    case 429:
      console.error('Rate limited. Retry after:', response.headers.get('Retry-After'));
      break;
    case 500:
      console.error('Server error:', error.message);
      break;
    default:
      console.error('Request failed:', error);
  }
  return;
}

const data = await response.json();
```

```python Python
import requests
from requests.exceptions import HTTPError

try:
    response = requests.post(
        'https://api.getcatalog.ai/v1/search',
        headers={
            'Content-Type': 'application/json',
            'x-api-key': 'YOUR_API_KEY'
        },
        json={'query': 'black dress'}
    )
    response.raise_for_status()
    data = response.json()
    
except HTTPError as e:
    if e.response.status_code == 401:
        print('Authentication failed')
    elif e.response.status_code == 429:
        print('Rate limited. Please wait before retrying.')
    elif e.response.status_code >= 500:
        print('Server error. Please retry.')
    else:
        print(f'Request failed: {e.response.json()}')
```

</CodeGroup>

### Retry with exponential backoff

For transient errors (429, 500), implement exponential backoff:

<CodeGroup>

```javascript JavaScript
async function requestWithRetry(url, options, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);
    
    if (response.ok) {
      return response.json();
    }
    
    if (response.status === 429 || response.status >= 500) {
      const delay = Math.pow(2, attempt) * 1000; // 1s, 2s, 4s
      console.log(`Retrying in ${delay}ms...`);
      await new Promise(resolve => setTimeout(resolve, delay));
      continue;
    }
    
    // Non-retryable error
    throw new Error(`Request failed: ${response.status}`);
  }
  
  throw new Error('Max retries exceeded');
}
```

```python Python
import time
import requests

def request_with_retry(url, headers, json_data, max_retries=3):
    for attempt in range(max_retries):
        response = requests.post(url, headers=headers, json=json_data)
        
        if response.ok:
            return response.json()
        
        if response.status_code in [429, 500, 502, 503, 504]:
            delay = (2 ** attempt)  # 1s, 2s, 4s
            print(f'Retrying in {delay}s...')
            time.sleep(delay)
            continue
        
        # Non-retryable error
        response.raise_for_status()
    
    raise Exception('Max retries exceeded')
```

</CodeGroup>

## Logical errors (200 OK responses)

Some endpoints return `200 OK` even when individual items fail. Always check the response body:

### Single product endpoint

```json
{
  "product": null,
  "meta": {
    "found": false,
    "url": "https://example.com/invalid-product",
    "outcome": "VENDOR_NOT_SUPPORTED"
  }
}
```

### Bulk products endpoint

```json
{
  "products": [
    {
      "url": "https://valid-store.com/product",
      "success": true,
      "product": { ... }
    },
    {
      "url": "https://invalid-store.com/product",
      "success": false,
      "product": null,
      "outcome": "VENDOR_NOT_SUPPORTED"
    }
  ],
  "meta": {
    "total_requested": 2,
    "total_successful": 1,
    "total_failed": 1
  }
}
```

<Tip>
Always check `meta.found` or `success` fields in responses, even when the HTTP status is 200.
</Tip>
